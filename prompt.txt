The project is event-based: every significant action performed by the scraper should immediately send an update back to the backend/frontend so the graph and UI stay live.

The flow should work like this:

* The frontend asks the user to create a scraper instance.
* Multiple scraper instances can be created.
* Each scraper instance must run in its own separate process and thread so that one scraper cannot affect another.
* Every scraper instance has its own:

  * API key
  * MongoDB URI
  * Database name
  * Proxy configuration

The backend in Go receives this data from the frontend and starts the scraper by calling the Python scraper class. All configuration values are passed into that scraper instance.

When the scraper starts:

1. A completely separate process is created for that scraper.
2. The model is initialized using the provided API key.
3. The initialized model should be cached or hashed so that it does not need to be recreated repeatedly, which helps avoid unnecessary rate limits.
4. The scraper generates search queries using DuckDuckGo, as it already does now.
5. All search results are placed into a queue.

The crawler should process links one parent at a time:

* Suppose there are 50 search result links.
* The scraper should fully explore the first link before moving to the second.
* Each parent link should maintain its own collections and document context.
* All child links discovered under that parent must remain connected to the same parent so that a proper graph structure can be built.

The entire system should remain event-driven. Every action taken by the scraper should immediately be sent back to the backend and to the frontend so the frontend can build and update a live graph.

The scraping flow for each page should work like this:

* Start with the first parent link.
* Use Crawl4AI to scrape everything on that page.
* Before processing the page, detect and handle ads:

  * If there is an on-click ad or redirect ad, click once if necessary, return to the original page, and do not store the ad page in the database.
  * If there is an overlay or popup ad, detect it, close it, and continue.
* Ads should never be stored as real nodes in the database or graph.

After the page is scraped:

* Collect all relevant information from the page.
* Extract all child links from that page.
* Send a summarized version of the page to the LLM, not the full raw page.
* The summary should include:

  * Main page details
  * All extracted child links
  * Any important metadata
* The LLM decides which child link should be visited next.

The graph should behave like a tree:

* The root node is the parent page.
* Every discovered link becomes a child node.
* When a child node is visited, the same process repeats:

  * Scrape the page
  * Extract links
  * Summarize the page
  * Ask the LLM which link to visit next
* Continue recursively, building a tree structure.

If the crawler reaches a player page:

* Scrape the player page.
* Handle player-specific ads such as:

  * “Wait 10 seconds”
  * “Skip now”
* The bot should wait if required, then press “Skip” automatically.
* Extract the final `.m3u8` stream URL from the player.
* Each player should have its own player ID.
* For each player ID, collect up to 5 stream URLs.
* Once 5 stream URLs are collected, store them in the database.

Traversal rules:

* Before visiting any URL, check whether it has already been visited.
* If it has already been visited, do not visit it again.
* The crawler stops only when:

  * No new links remain to be explored
  * Every node from root to leaf has already been visited

The frontend graph also needs improvement:

* The current graph is cluttered because edges overlap and intersect.
* Replace it with a clean tree-style graph layout.
* Parent-child relationships should be visually clear.
* Nodes should expand naturally like a hierarchy.

When the user clicks a node in the graph:

* Show a proper popup or side panel UI.
* Do not display raw JSON or dictionary output.
* The UI should clearly display:

  * Page title
  * URL
  * Whether the page was flagged
  * Suspicion score
  * Whether it is an ad
  * Whether it is a player page
  * Stream URLs
  * Child links
  * Any additional metadata
  * Crawl status
  * Time visited

The scraper should also recognize different kinds of ads:

* On-click redirect ads
* Overlay popup ads
* Fake “play” buttons
* Pages that redirect before reaching the real content

The bot should detect whether a page is an ad page, handle it correctly, and then continue to the actual page.

Additionally:

* Official websites should never be visited.
* Official domains should not be stored.
* The LLM should decide:

  * Which domains are official
  * Which domains are suspicious
  * Which privacy-related keywords matter
  * Which pages are likely pirated
  * Which VOD URLs are relevant
* Any hardcoded rules or constant lists for these decisions should be removed, since the LLM will handle those decisions dynamically.
